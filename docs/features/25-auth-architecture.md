# 25. 認証アーキテクチャ（ログインと外部連携の分離）— 方針メモ

[← 機能一覧に戻る](./README.md)

- ステータス: 📝 方針メモ（設計指針。別ログイン方式の実装は未着手）
- 対象: 共通（認証基盤）
- 関連: [01 Googleログイン](./01-google-login.md) / [02 Googleカレンダー連携](./02-google-calendar.md) / [db-schema.md](../db-schema.md)

## 採用するログイン方式（決定 2026-06-03）

- **Google ソーシャルログイン ＋ メール+パスワード の2方式のみ**を実装する（Apple / LINE は採用しない）。詳細は [open-decisions.md](../open-decisions.md) 決定3。
- メール+パスワードで登録したユーザーも、後から Google カレンダー連携を追加できる（下記アーキテクチャ）。

## 結論

**別のログイン方式（メール+パスワード等）を採用しても、後から Google カレンダー連携・他社連携は問題なく追加できる。**
ポイントは「認証（ログイン）」と「連携（外部APIの認可）」を別レイヤーとして分離すること。

## 認証と連携は別レイヤー

| 区分 | 役割 | 例 |
|---|---|---|
| **認証（ログイン）** | 「誰か」を確認しアカウントを特定 | メール+パスワード / Apple ID / 別の Google ログイン |
| **連携（認可）** | 外部 API を使う許可とトークン取得 | 「Google カレンダーと連携する」→ カレンダー権限取得 |

ログイン方法が何であっても、ログイン後に**別途**「Google カレンダーを連携する」フローを実行すれば、カレンダー連携・予定作成・Meet 発行はそのまま使える（Calendly 等と同じ方式）。

## キマルの現状と必要な変更

- 現状は **Google ログイン1回でカレンダー権限も同時取得**する作り（認証と連携が一体。[01](./01-google-login.md)）。
- 別ログインを足す場合は、**「カレンダー連携」を独立した OAuth フローに切り出す**だけで対応できる。
- **DB はすでに分離設計**:
  - `owners`（アカウント本体）と `google_connections`（`owner_id` 紐付けの連携トークン）が**別テーブル**（[db-schema.md](../db-schema.md)）。
  - → どのログイン方法でも、後から `owner_id` に対してカレンダー連携を追加できる構造。

## 連携時のポイント（仕様）

- 連携に使う Google アカウントは**ログイン用アカウントと別でもよい**（カレンダーだけ別 Google で連携も可）。
- 同じ仕組みで **Zoom や議事録アプリ等、他の外部連携も後付け可能**（1アカウントに複数連携）。連携対象は下記ロードマップ参照。
- トークンは暗号化保存・自動リフレッシュ（既存 `_lib/crypto.js` / `google_connections` を流用）。
- 連携状態（連携済み / 未連携）を相手管理・設定画面に表示する。

## 外部連携ロードマップ（連携対象）

連携する外部サービスは以下の3種類（この順で広げる）。

| 連携先 | 用途 | 状態 | 関連 |
|---|---|---|---|
| **Google** | カレンダー連携（空き判定・予定作成）/ Google Meet 自動発行 | ✅ 実装済（要設定） | [02](./02-google-calendar.md) / [09](./09-google-meet.md) |
| **Zoom** | Zoom ミーティングの自動発行 | 🔜 将来（無料・有料とも） | [06](./06-location-type.md) |
| **議事録アプリ類** | 会議の文字起こし・議事録の取り込み（tl;dv / Fireflies / Circleback / Meet純正API 等） | 🔜 後々（マスタープラン） | [23](./23-meeting-minutes.md) |

## 想定アーキテクチャ

```
[ログイン方式] ──(認証)──> owners（アカウント本体）
   ├ メール+パスワード
   ├ Google ログイン
   ├ Apple / LINE 等
                              │ owner_id
                              ▼
                    [外部連携テーブル群]（認可）
   ├ google_connections（Googleカレンダー / Meet）  … 実装済
   ├ zoom_connections（Zoom 自動発行）              … 将来
   └ minutes_connections（議事録アプリ連携）        … 後々（マスタープラン）
```

いずれの連携も `owner_id` に紐づくため、**ログイン方式に依存せず後付けで追加できる**。

## 残タスク（採用時）

- 「カレンダー連携」を独立 OAuth フロー化（`google-auth-callback` から連携処理を分離）。
- **メール+パスワード認証の基盤**: 登録・ログイン・パスワードハッシュ保存・パスワード再設定・メール確認。`owners` にパスワードハッシュ等のカラム追加。
- 設定画面に「Google カレンダー連携」ボタン（メール登録ユーザーが後から連携）。
- 設定画面に「Google カレンダー連携」ボタンと連携状態表示を追加。
- （将来）Zoom 連携テーブル・OAuth フロー追加（[06](./06-location-type.md)）。
- （後々）議事録アプリ連携（Webhook 受信／API）の追加（[23](./23-meeting-minutes.md)）。
